# KGQA Hierarchical: Technical Architecture Deep-Dive

This document provides an exhaustive, step-by-step breakdown of the Experiments 0 through 9 in the `kgqaHierarchical` project. It describes the precise data flow, neural architectures, and research rationale behind each design iteration.

---

## Experiment 0: Flat BERT Baseline
**Goal**: Establish a lower-bound for relation classification accuracy on CWQ.

### 1. How it Starts
The workflow begins in `train/exp0_flat_baseline.py`. It loads `train_questions_raw.pt` (strings) and `train_relations.pt` (tensors) from `data/processed_entity`.

### 2. Input Data
*   **Question**: Natural language string (e.g., *"Who was the director of Kismet?"*).
*   **Target**: A single Integer ID representing the 1st relation in the reasoning path (one of 916 classes).

### 3. Processing (Architecture)
1.  **Tokenization**: The question is tokenized using `BertTokenizer` (max length 128).
2.  **Encoding**: A `QuestionEncoder` (BERT-base-uncased) extracts the `[CLS]` token representation ($h \in \mathbb{R}^{768}$).
3.  **Classification Head**:
    *   `Linear(768 \to 1024)`
    *   `GELU` Activation
    *   `LayerNorm`
    *   `Linear(1024 \to 916)`
4.  **Loss**: `CrossEntropyLoss` is computed against the gold relation ID.

### 4. Why this was done
To determine if a standard, non-hierarchical classifier can solve complex questions. Since CWQ is multi-hop, a flat classifier usually fails because it only sees the "first" relation and ignores the rest of the path.

### 5. Output
A probability distribution over 916 Freebase relations.

## Experiment 1: Domain-Restricted Search
**Goal**: Prune the KG search space by predicting "Domains" instead of raw relations.

### 1. How it Starts
Launched via `train/exp1_domain_baseline.py`. It maps every relation (e.g., `film.film.directed_by`) to its domain prefix (`film`).

### 2. Input Data
*   **Question**: Natural language string.
*   **Target**: A Domain ID representing one of 69 high-level Freebase domains.

### 3. Processing (Architecture)
1.  **Encoder**: BERT-base-uncased (768).
2.  **Head**: `Linear(768 \to 1024) \to GELU \to Linear(1024 \to 69)`.
3.  **Inference (Searcher)**:
    *   The model predicts the Top-k domains.
    *   The `DomainRestrictedSearcher` iterates through the KG, but only allows relations that belong to those predicted domains.

### 4. Why this was done
Pruning the search space from 916 relations to ~10 relations (within a domain) significantly reduces the branching factor in physical KG traversal, improving execution speed.

### 5. Output
A 69-class probability distribution over domains.

## Experiment 2: Contrastive Path Discrimination (CPD)
**Goal**: Teach the model to distinguish between "similar but wrong" reasoning paths.

### 1. How it Starts
Launched via `train/exp2_cpd.py`. It uses a custom `PathDataset` that parses SPARQL queries to extract the *full reasoning chain*.

### 2. Input Data
*   **Anchor**: Question string.
*   **Positive Path**: A sequence of relation IDs extracted from the gold SPARQL.
*   **Negative Paths**: 4 "Hard Negatives" generated by swapping a single relation in the gold path with its nearest neighbor in embedding space.

### 3. Processing (Architecture)
1.  **Question Branch**: BERT \to `Linear(768 \to 512)` \to L2 Norm.
2.  **Path Branch**:
    *   `Embedding(916 \to 128)`
    *   `BiLSTM(128 \to 256)` \to Concatenated output (512) \to L2 Norm.
3.  **Contrastive Objective**: InfoNCE loss. It calculates the dot-product similarity between the Question Vector and the Path Vector.

### 4. Why this was done
Standard classification doesn't capture the "sequence" logic. CPD forces the model to understand the *entire path* and penalizes it if it survives on a path that is "close" but logically incorrect (e.g. `born_in` vs `lived_in`).

### 5. Output
A similarity score ranking the best path.

## Experiment 3: Progressive Constraint Tightening (PCT)
**Goal**: Build a hierarchical multi-task head for "self-aware" reasoning.

### 1. How it Starts
Launched via `train/exp3_pct.py`.

### 2. Input Data
*   **Question**: String.
*   **Targets**: Domain ID, Sub-domain ID, and Relation ID.

### 3. Processing (Architecture)
*   **Shared Backbone**: BERT + `Linear(768 \to 1024) \to GELU \to LayerNorm`.
*   **Multi-Task Heads**:
    1.  **Domain Head**: Coarse classification (69).
    2.  **Subdomain Head**: Mid-level classification (200).
    3.  **Relation Head**: Fine classification (916).
    4.  **Confidence Head**: A binary head (1) predicting the probability that the relation head is correct.

### 4. Why this was done
By training on hierarchical labels, the model learns features that are robust across different levels of abstraction. The **Confidence Head** is critical for later RL experiments, as it tells the agent when to trust its predictions.

### 5. Output
Four separate logits (Domain, Subdomain, Relation, Confidence).

---

## Experiment 4: Cross-Hop Coherent Planning (CHCP)
**Goal**: Replace independent hop classification with a joint Transformer planner.

### 1. How it Starts
Launched via `train/exp4_chcp.py`.

### 2. Input Data
*   **Question**: String.
*   **Target Path**: A fixed-length sequence of 4 relation IDs (padded with 0).

### 3. Processing (Architecture)
1.  **Encoder**: BERT-base.
2.  **Hop Slots**: The model defines 4 learnable vectors representing "First Hop", "Second Hop", etc.
3.  **Transformer Refinement**: 
    *   Input: `[Question_Proj + Hop1_Emb, Question_Proj + Hop2_Emb, ...]`
    *   A `TransformerEncoder` (2 layers, 4 heads) performs self-attention across these slots.
4.  **Joint Heads**: Predicts all 4 relations and all 4 "Stop" tokens simultaneously.
5.  **Coherence Loss**: Adds a penalty if $(r_k, r_{k+1})$ is an unlikely transition according to a learned transition matrix.

### 4. Why this was done
In multi-hop QA, the 2nd hop depends on the 1st hop. Sequential classifiers lose this context. The Transformer allows the "Second Hop" slot to "look back" at the "First Hop" slot to ensure the plan is logically consistent.

### 5. Output
A 4x916 matrix of relation probabilities.

## Experiment 4-RL: KG Execution Fine-tuning
**Goal**: Optimize the model for physical entity retrieval, not just label matching.

### 1. How it Starts
Launched via `train/exp4_rl.py`. Loads the pre-trained weights from Exp 4.

### 2. Input Data
*   **Question**: String.
*   **Knowledge Graph**: A physical subgraph in memory (subject-predicate-object triples).

### 3. Processing (Architecture)
1.  **Policy**: The CHCP model (Exp 4).
2.  **Sampling**: Instead of `argmax`, the agent samples a path from the probability distribution.
3.  **Execution**: The agent walks the KG using the sampled path.
4.  **Reward**: 
    *   +1.0 for success (reaching a gold answer entity).
    *   -0.1 for dead ends.
    *   -0.05 efficiency penalty (shorter paths are better).
5.  **Update**: REINFORCE algorithm updates the weights to maximize expected reward.

### 4. Why this was done
Supervised learning requires "Gold SPARQL," which is hard to collect. RL allows the model to learn from "weak supervision"—it just needs to know if the final answer was right or wrong.

## Experiment 6: Unified Adaptive-CHCP
**Goal**: Combine the best of Hierarchy (Exp 3) and Planning (Exp 4).

### 1. How it Starts
Launched via `train/exp6_unified.py`. 

### 2. Processing (Architecture)
It integrates the **Confidence Head** from Exp 3 into the **Transformer Planner** from Exp 4. The model now plans the entire path while simultaneously estimating how "sure" it is about the domain and the stop condition.

### 3. Why this was done
This is the "Integrated Foundation." It prepares the architecture for the scaling phase by creating a unified feature space for all reasoning tasks.

## Experiment 7: Escalation to RoBERTa-Large
**Goal**: Break the 65% Hits@1 barrier using massive pre-training.

### 1. How it Starts
Launched via `train/exp7_roberta.py`.

### 2. Input Data
Tokenized using `RobertaTokenizer` (max length 128).

### 3. Processing (Architecture)
*   **Backbone**: `roberta-large` (1024-dim hidden state, 1.5GB parameters).
*   **Transformer**: Scaled to 4 layers and 8 heads.
*   **Memory Tricks**: Uses **Gradient Accumulation** (4 steps) and Batch Size 4 to fit the large model into 8GB VRAM.
*   **Loss**: Multi-task CE on all heads.

### 4. Why this was done
RoBERTa-Large has a significantly better semantic understanding of rare words and complex syntax than BERT-base.

### 5. Output
State-of-the-art relation planning logits.

---

## Experiment 8: Contrastive RoBERTa (CPD-R)
**Goal**: Combine RoBERTa's capacity with CPD's discrimination.

### 1. How it Starts
Launched via `train/exp8_cpd_roberta.py`. Prerequisite: Exp 7 weights.

### 2. Processing (Architecture)
It applies the **Path-Contrastive Loss** from Exp 2 but uses a **Dynamic Adversarial sampler**. For every batch, it finds the most "confusing" negative path (the one the model currently ranks #2) and calculates InfoNCE loss to push it away.

### 3. Why this was done
Even RoBERTa can get confused by similar relations. CPD-R "sharpens" the decision boundaries, resulting in the high 72%+ accuracy scores.

## Experiment 9: RL Meta-Constraint Agent (RLMC)
**Goal**: Balance Accuracy and Efficiency via dynamic beam adjustment. This is the **Final SOTA Model**.

### 1. How it Starts
Launched via `train/exp9_rlmc.py`. It freezes the planning backbone (RoBERTa) and only trains the meta-policy.

### 2. Input Data
The hidden representations ($h \in \mathbb{R}^{512}$) from the Transformer planner for each hop.

### 3. Processing (Architecture)
1.  **State**: The Transformer representation of the current hop.
2.  **Meta-Policy Head**: A 4-action MLP classifier:
    *   **Action 0: TIGHT** (Use Top-1 beam).
    *   **Action 1: MEDIUM** (Use Top-5 beam).
    *   **Action 2: LOOSE** (Use Domain Fallback / Top-50).
    *   **Action 3: STOP** (End reasoning).
4.  **Reward**:
    *   +1.0 for choosing TIGHT and being correct (Efficiency Winner).
    *   +0.5 for MEDIUM and being correct.
    *   -1.0 for failing (regardless of beam width).
5.  **Training**: PPO (Proximal Policy Optimization).

### 4. Why this was done
Fixed-width search is slow. The RLMC Agent acts like a "manager"—it looks at the question's ambiguity and decides, "This is an easy 1st hop, use TIGHT," followed by, "The 2nd hop is ambiguous, use MEDIUM."

### 5. Output
A reasoning policy achieving **73.61% Hits@1** with significantly fewer KG lookups than a fixed-width search.

## 🛠️ System Utilities & Data Pipeline

To support these experiments, the repository contains several mission-critical utility scripts:

### 1. Data Construction (`data/build_supervision_with_entities.py`)
**Role**: The foundation of the supervision signal.
*   **Process**: It parses the raw CWQ JSON files and uses the `utils/sparql_parser.py` to extract the ground-truth reasoning paths from the SPARQL queries.
*   **Output**: It generates the `.pt` files (tensors) consumed by the training scripts, including `train_questions_raw.pt`, `train_relations.pt`, and `train_domains.pt`.

### 2. E2E Evaluation Engine (`eval/e2e_evaluate.py`)
**Role**: Standardized Path-Matching evaluation.
*   **Process**: It loads a saved checkpoint, runs the model over the `dev` or `test` set, and calculates **Hits@1**, **Hits@3**, and **MRR**.
*   **Why**: This provides a fast, non-execution-based metric to track model progress during training.

### 3. Physical Execution Eval (`eval/freebase_execution_eval.py`)
**Role**: The "Real World" validator.
*   **Process**: It takes the model's predicted relation path and physically traverses a local Knowledge Graph (loaded via `shared/kg_loader.py`).
*   **Why**: Path-matching is a proxy; the ultimate goal is to find the correct answer entity in the KG. This script provides the 73.61% Hits@1 SOTA metric.

### 4. SPARQL Parser (`utils/sparql_parser.py`)
**Role**: Semantic extraction.
*   **Process**: Uses regex and logic to pull triples out of complex CWQ SPARQL queries. It identifies the "Topic Entity" and the "Path" (sequence of predicates) that the model needs to learn.
